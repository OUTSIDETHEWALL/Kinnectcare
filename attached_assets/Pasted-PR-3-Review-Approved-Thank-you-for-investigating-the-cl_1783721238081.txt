Cleanup Review

My primary concern was the lifecycle of the temporary subscription.

Originally, cleanup relied on a future event (either another store update or the timeout) before the subscription was actually removed. While the maximum lifetime was bounded by the 90-second timeout, cleanup was not deterministic.

The revised implementation addresses that concern cleanly.

By introducing a cancelWait handle and calling it directly from the effect cleanup, the subscription, timer, and waiting Promise are now torn down immediately when the effect is cancelled.

That gives every exit path deterministic cleanup:

Successful member discovery
Timeout
Effect cancellation
Component unmount
Sign-out
Dependency change

The combination of the existing settled guard together with the explicit cancelWait teardown also provides good protection against race conditions during shutdown.

Architecture

I also agree with the decision to keep this repair inside the boot effect rather than modifying memberStore.fetchAll().

The store should remain responsible for retrieving member data.

The boot effect is responsible for deciding what the absence of a member row means for the location engine.

That separation of responsibility feels correct and minimizes regression risk for other consumers of the store.

Deployment

Because this change only modifies React Native / TypeScript application logic, this should be deployed as an OTA update rather than requiring a new Android build.

Expected Results

Following deployment, I expect:

Existing users with healthy member rows should experience no behavioral change.
Users encountering the member-row race during onboarding should recover automatically instead of silently remaining stopped.
Temporary /members failures should no longer permanently disable the location engine for the remainder of the session.
Legitimate cases where no member row exists should still stop the engine after the waiting period.
Regression risk should remain low because the normal startup path has not been modified.
